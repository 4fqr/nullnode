Chapter 04: Object-Oriented Programming

Creating Blueprints and Building Complex Systems

Part 1: What IS Object-Oriented Programming?

So far, you've been writing procedural code - step-by-step instructions that run from top to bottom: 

name = "Alice"
health = 100
damage = 20
print(f"{name} has {health} health")

But what if you're building a game with 50 characters? You'd need 50 name variables, 50 health variables, 50 damage variables... CHAOS!

The Blueprint Metaphor

Imagine you're a builder who needs to construct 100 houses. Do you:

  • Option A: Measure and plan each house individually from scratch (exhausting!)
  • Option B: Create ONE blueprint, then build 100 houses from that blueprint (smart!)

Object-Oriented Programming (OOP) is Option B for code:

  • You create a CLASS (the blueprint)
  • You build OBJECTS (the actual houses) from that class
  • Each object has the same structure but different details (same floor plan, different paint color)

Think of OOP as organizing code like you organize the real world:

  • Dogs are a CLASS - all dogs have name, breed, age
  • Your dog "Buddy" is an OBJECT - a specific dog with name="Buddy", breed="Golden Retriever", age=3
  • Cars are a CLASS - all cars have model, color, speed
  • Your car is an OBJECT - a specific car with model="Tesla", color="red", speed=0

The Cookie Cutter Metaphor

Another way to think about it:

  • CLASS = Cookie cutter (the shape/template)
  • OBJECTS = The actual cookies you make (each is a separate cookie, but all have the same shape)

You design the cookie cutter ONCE, then stamp out as many cookies as you need!

Part 2: Creating Your First Class

Let's create a Dog class (the blueprint) and then make actual dog objects from it:

Demo: Basic Class

# DEFINE the class (create the blueprint)
class Dog:
    def __init__(self, name, breed, age):
        self.name = name      # Attribute: the dog's name
        self.breed = breed    # Attribute: the dog's breed
        self.age = age        # Attribute: the dog's age
    
    def bark(self):
        print(f"{self.name} says: Woof! Woof!")
    
    def celebrate_birthday(self):
        self.age += 1
        print(f"{self.name} is now {self.age} years old!")

# CREATE objects (build actual dogs from the blueprint)
dog1 = Dog("Buddy", "Golden Retriever", 3)
dog2 = Dog("Max", "German Shepherd", 5)

# USE the objects
dog1.bark()                 # Output: Buddy says: Woof! Woof!
dog2.bark()                 # Output: Max says: Woof! Woof!
dog1.celebrate_birthday()  # Output: Buddy is now 4 years old!

Understanding the Parts

1. The Class Definition

class Dog:

This line says "I'm creating a new blueprint called Dog". Class names always start with a CAPITAL letter by convention.

2. The __init__ Method (Constructor)

def __init__(self, name, breed, age):
    self.name = name
    self.breed = breed
    self.age = age

__init__ is a special method that runs AUTOMATICALLY when you create a new object. Think of it as the "setup instructions" - it sets the initial values for your object.

  • self - Refers to "this specific object" (we'll explain more below)
  • name, breed, age - The information you need to provide when creating a dog
  • self.name = name - Store the name inside this specific dog object

3. Attributes (Properties)

self.name = name
self.breed = breed
self.age = age

Attributes are the characteristics/properties of an object. Every dog has these qualities stored inside it.

4. Methods (Actions)

def bark(self):
    print(f"{self.name} says: Woof!")

Methods are functions that belong to a class. They define what an object can DO. Dogs can bark, celebrate birthdays, etc.

What is "self"?

The Name Tag Metaphor

Imagine you're at a party with 10 people all named "Alex". When you say "Hey Alex!", how do they know WHICH Alex you mean?

In Python, when you have multiple dog objects, self is like pointing and saying "THIS specific dog right here, not the other dogs."

  • When you call dog1.bark(), inside the method self refers to dog1
  • When you call dog2.bark(), inside the method self refers to dog2

self is Python's way of saying "the object that called this method."

Part 3: Game Character System

Let's build something more practical - a video game character system:

Demo: Game Character Class

class Character:
    def __init__(self, name, health, attack_power):
        self.name = name
        self.health = health
        self.attack_power = attack_power
        self.is_alive = True
    
    def attack(self, target):
        """Attack another character."""
        if not self.is_alive:
            print(f"{self.name} is dead and can't attack!")
            return
        
        print(f"{self.name} attacks {target.name} for {self.attack_power} damage!")
        target.take_damage(self.attack_power)
    
    def take_damage(self, damage):
        """Reduce health when taking damage."""
        self.health -= damage
        
        if self.health <= 0:
            self.health = 0
            self.is_alive = False
            print(f"{self.name} has been defeated!")
        else:
            print(f"{self.name} has {self.health} health remaining.")
    
    def heal(self, amount):
        """Restore health."""
        if not self.is_alive:
            print(f"{self.name} is dead and can't be healed!")
            return
        
        self.health += amount
        print(f"{self.name} healed for {amount}! Health: {self.health}")
    
    def show_stats(self):
        """Display character information."""
        status = "ALIVE" if self.is_alive else "DEAD"
        print(f"\n--- {self.name} ---")
        print(f"Health: {self.health}")
        print(f"Attack Power: {self.attack_power}")
        print(f"Status: {status}")

# Create characters
hero = Character("Aragorn", 100, 25)
enemy = Character("Orc", 60, 15)

# Show initial stats
hero.show_stats()
enemy.show_stats()

# Battle!
print("\n=== BATTLE START ===")
hero.attack(enemy)
enemy.attack(hero)
hero.attack(enemy)
hero.attack(enemy)

# Show final stats
print("\n=== BATTLE END ===")
hero.show_stats()
enemy.show_stats()

Part 4: Inheritance - The Family Tree

The Family Tree Metaphor

Think about how traits are passed down in families:

  • You inherit traits from your parents (hair color, height genes)
  • You also have your OWN unique traits (your personality, skills)
  • You're not EXACTLY like your parents, but you share some characteristics

Inheritance in OOP works the same way:

  • A child class inherits attributes and methods from a parent class
  • The child can ADD its own unique attributes and methods
  • The child can OVERRIDE parent methods to change behavior

Let's expand our game to have different character TYPES (Warrior, Mage, Archer):

Demo: Inheritance in Action

# PARENT class - shared by all characters
class Character:
    def __init__(self, name, health, attack_power):
        self.name = name
        self.health = health
        self.attack_power = attack_power
    
    def attack(self):
        print(f"{self.name} attacks for {self.attack_power} damage!")

# CHILD class - inherits from Character, adds unique abilities
class Warrior(Character):
    def __init__(self, name, health, attack_power, armor):
        super().__init__(name, health, attack_power)  # Call parent constructor
        self.armor = armor  # Unique to Warrior
    
    def shield_bash(self):
        """Special ability only Warriors have."""
        damage = self.attack_power + self.armor
        print(f"{self.name} uses Shield Bash for {damage} damage!")

class Mage(Character):
    def __init__(self, name, health, attack_power, mana):
        super().__init__(name, health, attack_power)
        self.mana = mana  # Unique to Mage
    
    def fireball(self):
        """Special ability only Mages have."""
        if self.mana >= 20:
            self.mana -= 20
            damage = self.attack_power * 2
            print(f"{self.name} casts Fireball for {damage} damage! (Mana: {self.mana})")
        else:
            print(f"{self.name} doesn't have enough mana!")

class Archer(Character):
    def __init__(self, name, health, attack_power, arrows):
        super().__init__(name, health, attack_power)
        self.arrows = arrows  # Unique to Archer
    
    def multi_shot(self):
        """Special ability only Archers have."""
        if self.arrows >= 3:
            self.arrows -= 3
            damage = self.attack_power * 3
            print(f"{self.name} fires Multi-Shot for {damage} damage! (Arrows: {self.arrows})")
        else:
            print(f"{self.name} doesn't have enough arrows!")

# Create different character types
warrior = Warrior("Thorgrim", 150, 20, 50)
mage = Mage("Gandalf", 80, 15, 100)
archer = Archer("Legolas", 100, 18, 50)

# All can use basic attack (inherited from Character)
warrior.attack()
mage.attack()
archer.attack()

print("\n=== SPECIAL ABILITIES ===")

# Each has unique special ability
warrior.shield_bash()
mage.fireball()
archer.multi_shot()

Output:

Thorgrim attacks for 20 damage!
Gandalf attacks for 15 damage!
Legolas attacks for 18 damage!

=== SPECIAL ABILITIES ===
Thorgrim uses Shield Bash for 70 damage!
Gandalf casts Fireball for 30 damage! (Mana: 80)
Legolas fires Multi-Shot for 54 damage! (Arrows: 47)

Benefits of Inheritance

  • Code reuse: Don't repeat the same code in every class
  • Organization: Group similar objects under a parent class
  • Extensibility: Easy to add new character types without rewriting everything

Think of it like a company: All employees share basic traits (name, ID, salary), but Managers, Developers, and Designers each have unique responsibilities and tools.

Part 5: Encapsulation - Protecting Data

The Bank Vault Metaphor

Imagine a bank:

  • Your money is stored in a vault (protected, you can't just walk in and grab it)
  • You interact through a teller (safe, controlled access)
  • The teller can deposit money (controlled way to add)
  • The teller can withdraw money (controlled way to remove)

Encapsulation is the same idea for code:

  • Hide sensitive data inside the object (the vault)
  • Provide controlled methods to interact with that data (the teller)
  • Prevent direct access that could break things

Demo: Bank Account with Encapsulation

class BankAccount:
    def __init__(self, account_holder, initial_balance=0):
        self.account_holder = account_holder
        self.__balance = initial_balance  # __ makes it PRIVATE
        self.__transaction_history = []
    
    def deposit(self, amount):
        """Safely add money."""
        if amount > 0:
            self.__balance += amount
            self.__transaction_history.append(f"Deposited: ${amount}")
            print(f"Deposited: ${amount}")
        else:
            print("Invalid deposit amount!")
    
    def withdraw(self, amount):
        """Safely remove money."""
        if 0 < amount <= self.__balance:
            self.__balance -= amount
            self.__transaction_history.append(f"Withdrew: ${amount}")
            print(f"Withdrew: ${amount}")
        else:
            print("Insufficient funds or invalid amount!")
    
    def get_balance(self):
        """Safely check balance."""
        return self.__balance
    
    def show_transactions(self):
        """Show transaction history."""
        print(f"\n=== Transaction History for {self.account_holder} ===")
        for transaction in self.__transaction_history:
            print(transaction)
        print(f"Current Balance: ${self.__balance}\n")

# Create account
account = BankAccount("Alice", 1000)

# Use SAFE methods
account.deposit(500)
account.withdraw(200)
account.withdraw(5000)  # This will be rejected!

# Check balance SAFELY
print(f"Balance: ${account.get_balance()}")
account.show_transactions()

# Try to access private data directly (BAD! Will fail or cause issues)
# account.__balance = 999999999  # This won't work as expected!

Why Encapsulation Matters

Without encapsulation, anyone could do: 

account.balance = -1000000  # Negative balance? That makes no sense!
account.balance = "banana"  # Balance is text? BROKEN!

With encapsulation using methods (deposit(), withdraw()), you control HOW data changes, preventing invalid states!
