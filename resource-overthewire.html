OverTheWire: Complete Walkthrough

Every level explained in extreme detail for absolute beginners

What is OverTheWire?

OverTheWire is a collection of "wargames" - security challenges that teach you hacking skills through practice. Think of it like a video game, but instead of fighting monsters, you're solving Linux puzzles and learning real hacking techniques.

Bandit is the first and most beginner-friendly wargame. It teaches you Linux command line basics, which is the foundation of ALL hacking.

üìö What You'll Learn:

  • ‚úÖ Linux command line basics (ls, cd, cat, grep)
  • ‚úÖ File permissions and hidden files
  • ‚úÖ SSH connections and remote servers
  • ‚úÖ Finding and reading files in Linux
  • ‚úÖ Data encoding (base64, hex, rot13)
  • ‚úÖ Compression and archives
  • ‚úÖ Network connections and ports

üöÄ Getting Started

Step 1: What is SSH?

SSH (Secure Shell) is a way to control a computer remotely through the command line. Think of it like remote desktop, but instead of a graphical interface, you get a terminal where you type commands.

When you "SSH into" a server, you're connecting to another computer over the internet and controlling it with text commands.

Step 2: Open Your Terminal

On Windows: Use PowerShell, Command Prompt, or install WSL (Windows Subsystem for Linux)

On Mac/Linux: Open the Terminal app (usually in Applications ‚Üí Utilities)

Step 3: Understand the Connection Command

ssh bandit0@bandit.labs.overthewire.org -p 2220

Let's break this down word by word:

  • ssh - The command to start an SSH connection
  • bandit0 - The username you're logging in as
  • @ - Separator between username and server address
  • bandit.labs.overthewire.org - The server's address (like a website URL)
  • -p 2220 - The port number (like an apartment number in a building)

Level 0 ‚Üí 1

Reading Files with cat

üéØ Goal:

The password for Level 1 is stored in a file called readme in your home directory. Find and read it.

üí° Concepts You Need to Know:

What is a "home directory"?
When you log into a Linux system, you start in your "home directory" - think of it as your personal folder. It's like "My Documents" on Windows. When you first connect via SSH, you're automatically placed here.

What does "cat" do?
The cat command (short for "concatenate") reads files and displays their content in the terminal. Think of it as "open and show me what's inside this file."

What is "ls"?
The ls command (short for "list") shows you all files and folders in your current location. It's like opening File Explorer on Windows.

üìù Step-by-Step Solution:

Step 1: See what files are here

# Type this command:

ls

# You'll see output like:

readme

Perfect! There's a file called "readme" here. This is what we're looking for.

Step 2: Read the file

# Type this command:

cat readme

# You'll see the password printed on screen:

NH2SXQwcBdpmTEzi3bvBHMM9H66vVXjL

This long string of random characters is the password for Level 1! Copy it - you'll need it to log into the next level.

Step 3: Disconnect and move to Level 1

Now connect to Level 1 using the password you just found:

ssh bandit1@bandit.labs.overthewire.org -p 2220

# When prompted, paste the password you copied

üß† What You Learned:

  • ls - Lists files in your current directory
  • cat filename - Reads and displays file content
  • exit - Disconnects from SSH session
  • How to navigate between levels using the password you find
Level 1 ‚Üí 2

Special Characters in Filenames

üéØ Goal:

The password is stored in a file called - (just a dash/hyphen). Read it.

üí° The Problem:

In Linux, the dash (-) is a special character. It's normally used to indicate options/flags for commands. For example, ls -la means "run ls with the options l and a".

If you try cat -, the system thinks you're trying to give cat an option, not read a file named "-". This is confusing!

Solution: We need to tell the system "I mean a FILE called dash, not an option." We do this by specifying the FULL PATH to the file.

üí° Understanding Paths:

What is a "path"?
A path is the address of a file. Like how "123 Main Street, Apartment 4B" is a full address, a file path tells the computer exactly where a file is.

Two types of paths:
1. Absolute path: The FULL address starting from root. Example: /home/bandit1/myfile.txt
2. Relative path: The address relative to where you are now. Example: ./myfile.txt

The ./ means "in the current directory". It's like saying "in this folder I'm in right now."

üìù Step-by-Step Solution:

Step 1: Verify the file exists

ls

# Output:

-

Step 2: Read the file using its relative path

# This WON'T work:

cat -

# This WILL work:

cat ./-

# Output (the password):

rRGizSaX8Mk1RTb1CNQoXTcYZWU6lgzi

By adding ./ before the dash, we're saying "cat, please read the file named dash in my current directory" instead of "cat, I'm giving you an option."

üß† What You Learned:

  • Some characters are special in Linux (like -)
  • ./ means "current directory"
  • Using ./filename tells Linux you mean a file, not a command option
  • Alternative: You can also use cat < - (input redirection)

üìö Learning Pattern Established!

You now understand the fundamental pattern of Bandit:

  5. Use that password to access the next level

Each level teaches you a NEW Linux concept or command. The levels get progressively harder, building on what you've learned. Keep going - you're doing great! üéâ

In Linux, files starting with a dot (.) are hidden. When you run ls, you WON'T see them! Think of it like invisible ink - the files exist, but you need special glasses to see them.

Why hide files? Configuration files (like .bashrc, .ssh) are hidden so they don't clutter your view.

  2. You'll see a directory called "inhere":

  3. List hidden files in this directory:
    ls -la
    You'll see a file like .hidden
  4. Read it:

üí° What is "Human-Readable"?

Computers store TWO types of files:
1. Text files - You can read them (like .txt, .html, .py)

The file command tells you what type a file is!

üéØ Level 5 ‚Üí 6: Finding Files by Size & Type

Goal: Find a file that is:
  • Human-readable
  • 1033 bytes in size
  • Not executable

üí° The Find Command - Your Swiss Army Knife

find is like a super-powered search. Instead of just searching by name, you can search by: size, type, permissions, date modified, and more!

üìù Solution

  1. Navigate to inhere:
    cd inhere
  2. Use find with multiple conditions:
    find . -type f -size 1033c ! -executable
    Let's break this down:
    • . = search in current directory
    • -type f = only files (not directories)
    • -size 1033c = exactly 1033 bytes (c=characters/bytes)
    • ! -executable = NOT executable (the ! means NOT)
  3. It will show you the path, then read it:
    cat ./maybehere07/.file2 (or whatever path it found)
‚úÖ What You Learned:
  • find command with multiple filters
  • -size flag (c=bytes, k=kilobytes, M=megabytes)
  • -type f for files only
  • ! operator for negation (NOT)
  • How to combine multiple search criteria

üéØ Level 6 ‚Üí 7: Searching the Entire System

Goal: Find a file ANYWHERE on the server that is:
  • Owned by user bandit7
  • Owned by group bandit6
  • 33 bytes in size

üí° Understanding File Ownership

Every file in Linux has TWO owners:
1. User owner - The person who created it
2. Group owner - A group of users who can access it

This is like a document being owned by "John" but shared with the "Engineering Team".

üìù Solution

  1. Search from root directory:
    find / -user bandit7 -group bandit6 -size 33c 2>/dev/null
    Breaking it down:
    • / = start from root (search EVERYWHERE)
    • -user bandit7 = owned by bandit7
    • -group bandit6 = group is bandit6
    • -size 33c = 33 bytes
    • 2>/dev/null = hide "Permission denied" errors
  2. You'll see something like:
    /var/lib/dpkg/info/bandit7.password
  3. Read it:
    cat /var/lib/dpkg/info/bandit7.password
üí° Pro Tip: 2>/dev/null is your best friend!
2 = error messages (stderr)
>/dev/null = send to the void (discard them)
Without this, your screen would be flooded with "Permission denied" messages!
‚úÖ What You Learned:
  • File ownership (user and group)
  • -user and -group flags
  • Searching from root directory /
  • 2>/dev/null to suppress errors
  • How to find files system-wide

üìù Level 7 ‚Üí 8: Finding Text in Large Files with grep

üéØ GOAL:

Password is stored in data.txt next to word "millionth"
ssh bandit7@bandit.labs.overthewire.org -p 2220

üìö Concepts You Need to Know

What is grep?

grep = "Global Regular Expression Print". Think of it like Ctrl+F in a text editor, but WAY more powerful. It searches INSIDE files for specific text patterns.

Real-world analogy: Imagine you have a 1000-page book and need to find the word "millionth". You could read every page (would take forever), OR you could use grep which instantly finds it for you!

‚úÖ Step-by-Step Solution

  1. SSH into bandit7
  2. Check what's in the directory:
    ls -la
    You'll see data.txt - a large file with many lines
  3. Try reading it normally (bad idea):
    cat data.txt
    Whoa! Thousands of lines scroll past. This is NOT the way!
  4. Use grep to find "millionth":
    grep "millionth" data.txt
    Expected output:
    millionth [PASSWORD HERE]
‚úÖ What You Learned:
  • grep searches INSIDE files for text
  • It's MUCH faster than reading manually
  • Syntax: grep "search term" filename
  • grep shows the entire line containing the match
üí° grep Power Tips:
  • grep -i "word" file - Case-insensitive (finds Word, WORD, word)
  • grep -n "word" file - Show line numbers
  • grep -v "word" file - Show lines that DON'T match
  • grep -r "word" /path/ - Search recursively in all files

üî¢ Level 8 ‚Üí 9: Finding Unique Lines with sort and uniq

üéØ GOAL:

Password is the only line of text that occurs only once in data.txt
ssh bandit8@bandit.labs.overthewire.org -p 2220
Password: (from previous level)

üìö Concepts You Need to Know

What are sort and uniq?

sort: Arranges lines alphabetically (A-Z). Like sorting a deck of cards.
uniq: Removes duplicate CONSECUTIVE lines. But it ONLY works on sorted data!

Why sort first? Because uniq only removes duplicates that are next to each other. If you have: [apple, banana, apple], uniq won't remove the second apple because they're not consecutive. But if you sort first: [apple, apple, banana], NOW uniq works!

The pipe (|): This is the MAGIC of Linux! The | symbol takes the OUTPUT of one command and feeds it as INPUT to another command. It's like a production line in a factory!

‚úÖ Step-by-Step Solution

  1. Look at the file:
    cat data.txt
    You'll see many lines, most appearing multiple times
  2. Sort the file first:
    sort data.txt
    Now identical lines are grouped together
  3. Use uniq to find unique lines:
    sort data.txt | uniq -u
    Breaking this down:
    ‚Ä¢ sort data.txt - Sort all lines
    ‚Ä¢ | - Pipe (send output to next command)
    ‚Ä¢ uniq -u - Show only unique lines (the -u flag means "unique only")
  4. The password appears! That's the line that only occurs once.
‚úÖ What You Learned:
  • sort arranges lines alphabetically
  • uniq removes duplicate consecutive lines
  • uniq -u shows ONLY lines that appear once
  • | (pipe) chains commands together
  • Always sort BEFORE using uniq!
üí° More uniq Tricks:
  • uniq -c - Count how many times each line appears
  • uniq -d - Show only duplicate lines
  • sort | uniq - Remove all duplicates (keep one copy)

üî§ Level 9 ‚Üí 10: Extracting Text from Binary Files

üéØ GOAL:

Password is in data.txt, in one of the few human-readable strings, preceded by several '=' characters
ssh bandit9@bandit.labs.overthewire.org -p 2220

üìö Concepts You Need to Know

What is a binary file?

Most files are either text files (you can read them with cat) or binary files (compiled programs, images, etc. that look like gibberish if you cat them).

What is strings? The strings command extracts ONLY the human-readable text from binary files. It ignores all the binary garbage and shows you readable words!

Real-world analogy: Imagine a book where 99% of pages are in alien language, but 1% have English sentences. strings automatically finds and shows you ONLY the English sentences!

‚úÖ Step-by-Step Solution

  1. Check the file type:
    file data.txt
    Output: data.txt: data (binary file, not text!)
  2. Try to read it normally (fails):
    cat data.txt
    Your terminal shows garbage characters! ÔøΩÔøΩÔøΩ This is binary data.
  3. Extract readable strings:
    strings data.txt
    Now you see ONLY human-readable text!
  4. Search for lines with '==' characters:
    strings data.txt | grep "=="
    Expected output:
    ========== password
    ========== [THE PASSWORD]
‚úÖ What You Learned:
  • strings extracts readable text from binary files
  • Binary files contain non-text data (compiled code, images, etc.)
  • You can pipe strings output to grep for filtering
  • Combining commands with | is incredibly powerful
üí° strings Options:
  • strings -n 10 file - Show only strings of 10+ characters
  • strings -a file - Scan entire file (default only scans certain sections)
  • strings -o file - Show byte offset of each string

üîê Level 10 ‚Üí 11: Base64 Decoding

üéØ GOAL:

Password is in data.txt, which contains base64 encoded data
ssh bandit10@bandit.labs.overthewire.org -p 2220
Password: (from previous level)

üìö Concepts You Need to Know

What is Base64?

Base64 is an encoding (NOT encryption!) that converts binary data into ASCII text. It uses only letters (A-Z, a-z), numbers (0-9), and two symbols (+, /).

Why use it? Email and old systems can only handle text, not binary data. Base64 lets you send ANY file (images, programs) as text!

Example:
Original: Hello
Base64: SGVsbG8=
Notice the = at the end? That's padding! Base64 often ends with = or ==

‚ö†Ô∏è Important: Base64 is NOT secure! It's encoding, not encryption. Anyone can decode it instantly. It's just a way to represent data, not hide it!

‚úÖ Step-by-Step Solution

  1. Read the file:
    cat data.txt
    You'll see something like: VGhlIHBhc3N3b3JkIGlzIDZ6UGV6aUxkUjJSS05kTllGTmI2blZDS3pwaGxYSEJNCg==
  2. Notice the characteristics:
    • Only letters, numbers, +, /, and =
    • Ends with == (padding)
    • This is definitely Base64!
  3. Decode it:
    base64 -d data.txt
    Breaking it down:
    ‚Ä¢ base64 - The base64 command
    ‚Ä¢ -d - Decode flag (opposite of encode)
    ‚Ä¢ data.txt - The file to decode
  4. Output:
    The password is [PASSWORD HERE]
‚úÖ What You Learned:
  • Base64 is an encoding method (not encryption!)
  • base64 -d file decodes base64
  • base64 file encodes to base64
  • Base64 uses A-Z, a-z, 0-9, +, / and = for padding
  • Encoding ‚â† Encryption (base64 provides NO security)
üí° Base64 in the Real World:
  • Email attachments: Files are base64 encoded in email source
  • Data URLs: Images in HTML/CSS (data:image/png;base64,...)
  • JWTs: JSON Web Tokens use base64 for headers/payloads
  • Malware: Attackers sometimes base64-encode payloads to avoid detection

üîÑ Level 11 ‚Üí 12: ROT13 Cipher (Letter Rotation)

üéØ GOAL:

Password in data.txt where all lowercase (a-z) and uppercase (A-Z) letters have been rotated by 13 positions
ssh bandit11@bandit.labs.overthewire.org -p 2220
Password: (from previous level)

üìö Concepts You Need to Know

What is ROT13?

ROT13 = "Rotate by 13 places". It's a simple substitution cipher where each letter is replaced by the letter 13 positions after it in the alphabet.

The alphabet has 26 letters. So rotating by 13 means:
A ‚Üí N, B ‚Üí O, C ‚Üí P, ... M ‚Üí Z, N ‚Üí A, O ‚Üí B, ... Z ‚Üí M

Cool trick: ROT13 is its own inverse! If you ROT13 something twice, you get the original back:
Hello ‚Üí Uryyb ‚Üí Hello

The tr command: "translate" - it replaces characters. Syntax: tr 'old chars' 'new chars'

‚úÖ Step-by-Step Solution

  1. Read the encoded file:
    cat data.txt
    You'll see: Gur cnffjbeq vf 7k16WgUl0m... (gibberish!)
  2. Use tr to decode ROT13:
    cat data.txt | tr 'A-Za-z' 'N-ZA-Mn-za-m'
    Let's break this down character by character:
    • cat data.txt - Read the file
    • | - Pipe the output
    • tr - Translate command
    • 'A-Za-z' - OLD characters (all letters)
    • 'N-ZA-Mn-za-m' - NEW characters (rotated by 13)
  3. Understanding the rotation string:
    'N-ZA-M' means: N,O,P,Q,R,S,T,U,V,W,X,Y,Z,A,B,C,D,E,F,G,H,I,J,K,L,M
    This is the alphabet shifted by 13 positions!
  4. Output:
    The password is [PASSWORD]
‚úÖ What You Learned:
  • ROT13 is a Caesar cipher (shift cipher)
  • tr translates/replaces characters
  • Syntax: tr 'from' 'to'
  • A-Z notation represents a range of characters
  • ROT13 is symmetric (encode = decode)
üí° More tr Examples:
  • tr 'a-z' 'A-Z' - Convert lowercase to uppercase
  • tr -d 'a' - Delete all 'a' characters
  • tr -s ' ' - Squeeze repeated spaces into one
  • tr ' ' '_' - Replace spaces with underscores

üì¶ Level 12 ‚Üí 13: Multiple Compression Formats (The Nested Doll)

üéØ GOAL:

Password is in data.txt, which is a hexdump of a file that has been compressed multiple times
ssh bandit12@bandit.labs.overthewire.org -p 2220
Password: (from previous level)

‚ö†Ô∏è WARNING: This Level is TOUGH!

This is the first HARD level! The file is compressed with gzip, then bzip2, then tar, then gzip again, etc. - like Russian nesting dolls! You'll decompress, check file type, decompress again, repeat... about 8-10 times!

üìö Concepts You Need to Know

Compression Formats:

  • gzip (.gz): Common Linux compression. Decompress with gzip -d file.gz or gunzip file.gz
  • bzip2 (.bz2): Better compression than gzip. Decompress with bzip2 -d file.bz2 or bunzip2 file.bz2
  • tar (.tar): Archive format (not compression!). Think of it as a "folder" that bundles files. Extract with tar -xf file.tar
  • Hexdump: Binary data represented as hexadecimal text. Reverse with xxd -r

Key command: file filename - tells you what TYPE of file you're dealing with!

‚úÖ Step-by-Step Solution

  1. Create a working directory:
    mkdir /tmp/mywork && cd /tmp/mywork
    Why /tmp? You have write permission there! Your home directory is read-only.
  2. Copy the file:
    cp ~/data.txt .
    The . means "current directory"
  3. Convert from hexdump to binary:
    xxd -r data.txt > data.bin
    Breaking this down:
    ‚Ä¢ xxd -r - Reverse hexdump (convert hex to binary)
    ‚Ä¢ data.txt - Input (hex dump)
    ‚Ä¢ > data.bin - Output redirect to new file
  4. Check what you have:
    file data.bin
    Output: data.bin: gzip compressed data
    Aha! It's gzip!
  5. Decompress gzip (rename first):
    mv data.bin data.gz
    gzip -d data.gz
    This creates data (without .gz)
  6. Check again:
    file data
    Output: data: bzip2 compressed data
    Now it's bzip2!
  7. Decompress bzip2:
    mv data data.bz2
    bzip2 -d data.bz2
  8. Keep checking and decompressing:
    You'll encounter (in various orders):
    • file data ‚Üí gzip ‚Üí mv data data.gz && gzip -d data.gz
    • file data ‚Üí bzip2 ‚Üí mv data data.bz2 && bzip2 -d data.bz2
    • file data ‚Üí tar archive ‚Üí tar -xf data
  9. Repeat steps 8 about 8-10 times until:
    file data
    Output: data: ASCII text
    Finally! It's text!
  10. Read it:
    cat data
    The password is [PASSWORD]
‚úÖ What You Learned:
  • xxd -r converts hexdump to binary
  • gzip -d decompresses gzip files
  • bzip2 -d decompresses bzip2 files
  • tar -xf extracts tar archives
  • ALWAYS use file command to check file type!
  • Files can be compressed multiple times (nested)
üí° Quick Reference:

file data ‚Üí Check type

xxd -r hex.txt data ‚Üí Hex to binary

gzip -d file.gz ‚Üí Decompress gzip

bzip2 -d file.bz2 ‚Üí Decompress bzip2

tar -xf file.tar ‚Üí Extract tar

mv old new ‚Üí Rename file

üîë Level 13 ‚Üí 14: SSH Private Keys (Password-less Login)

üéØ GOAL:

Password for next level is stored in /etc/bandit_pass/bandit14 (you can't read it yet!)
You have an SSH private key in your home directory to log in as bandit14
ssh bandit13@bandit.labs.overthewire.org -p 2220

üìö Concepts You Need to Know

What is SSH Key Authentication?

Usually you SSH with a password. But there's a MORE SECURE way: SSH keys!

How it works:
‚Ä¢ You have TWO keys: private key (secret, keep it safe!) and public key (can share with anyone)
‚Ä¢ The public key is placed on the server you want to access
‚Ä¢ When you connect, you present your private key
‚Ä¢ The server checks: "Does this private key match the public key I have?" If yes ‚Üí you're in!

Real-world analogy: The public key is a lock, the private key is the key that opens it. You can give out many copies of the lock (public key), but only YOUR key (private key) can open them!

The -i flag: Tells SSH "use THIS specific private key file"

‚úÖ Step-by-Step Solution

  1. List files in home directory:

What is SSL/TLS?

SSL (Secure Sockets Layer) and TLS (Transport Layer Security) encrypt your network traffic so nobody can spy on it!

Real-world analogy: Regular netcat is like sending a postcard (anyone can read it). SSL is like putting it in a locked box with a code only you and the recipient know!

When you see HTTPS: That's HTTP + SSL/TLS. The padlock üîí in your browser means encryption is active!

OpenSSL: The Swiss Army knife for SSL/TLS. It can:

  • Create encrypted connections
  • Generate certificates
  • Test SSL configurations
  • Encrypt/decrypt data

‚úÖ Step-by-Step Solution

    ‚Ä¢ s_client - SSL client mode (connect to server)
    ‚Ä¢ -connect localhost:30001 - Where to connect
    ‚Ä¢ -ign_eof - Ignore end-of-file (keeps connection open)
  3. You'll see lots of certificate info scroll by:

    Certificate chain
    SSL handshake has read 1234 bytes
    ...

  4. Paste the password and press Enter
  5. Response:
    Correct!
    [BANDIT16 PASSWORD]
‚úÖ What You Learned:

  • openssl s_client creates encrypted connections
  • HTTPS = HTTP + SSL/TLS
  • Certificates verify server identity
  • Port 443 is standard for HTTPS
üí° OpenSSL Power Commands:
  • openssl s_client -connect site.com:443 - Test HTTPS connection

üîç Level 16 ‚Üí 17: Port Scanning (Finding Open Ports)

üéØ GOAL:

üìö Concepts You Need to Know

What is Port Scanning?

Port scanning = checking which ports are OPEN (accepting connections) on a computer. It's like trying every door in a building to see which ones are unlocked!

Nmap: The KING of port scanners! Used by hackers, pentesters, and sysadmins worldwide. It can:

  • Find open ports
  • Detect what services are running (SSH? Web server?)
  • Identify operating system
  • Check for vulnerabilities

‚úÖ Step-by-Step Solution

  1. Scan the port range:
    nmap -p 31000-32000 localhost
    Breaking this down:
    ‚Ä¢ nmap - Network mapper (port scanner)
    ‚Ä¢ -p 31000-32000 - Port range to scan
    ‚Ä¢ localhost - Target (this machine)
  2. You'll see output like:
    PORT STATE SERVICE
    31046/tcp open unknown
    31518/tcp open unknown
    31691/tcp open unknown
    31790/tcp open unknown
    31960/tcp open unknown

  3. Test each port with SSL:
    Try each open port:
    openssl s_client -connect localhost:31046 -ign_eof

    Paste the current password
    You'll get back an SSH PRIVATE KEY, not a password!
  5. Save the private key:
    mkdir /tmp/mykey && cd /tmp/mykey
    nano sshkey.private
    Paste the entire key (from BEGIN to END)
    Save with Ctrl+O, Enter, Ctrl+X
  6. Set permissions:

    SSH keys MUST have strict permissions!
  7. Use it to log in as bandit17:
    ssh -i sshkey.private bandit17@localhost -p 2220
‚úÖ What You Learned:

  • SSH keys can be used instead of passwords
üí° Nmap Scan Types:

üìö Concepts You Need to Know

What is diff?

diff compares two files line-by-line and shows the differences. It's like "track changes" in Microsoft Word!

Output format:
‚Ä¢ Lines starting with < = in first file only
‚Ä¢ Lines starting with > = in second file only
‚Ä¢ Lines with --- = separator
‚Ä¢ Numbers like 42c42 = line 42 changed

Real-world uses:

  • Comparing code versions (Git uses diff internally!)
  • Finding configuration changes
  • Detecting file tampering
  • Code reviews

‚úÖ Step-by-Step Solution

  1. List the files:
    ls -la
    You see: passwords.old and passwords.new
  2. Check file sizes:

  3. Use diff to compare:
    diff passwords.old passwords.new

    42c42
    < [OLD PASSWORD]
    ---
    > [NEW PASSWORD]
  4. Understanding the output:
    ‚Ä¢ 42c42 - Line 42 changed
    ‚Ä¢ < [OLD PASSWORD] - What was in passwords.old
    ‚Ä¢ > [NEW PASSWORD] - What's now in passwords.new
    The NEW password (after >) is the answer!
‚úÖ What You Learned:

  • < means "in first file"
  • > means "in second file"
  • diff is perfect for comparing versions
  • wc -l counts lines in files
üí° diff Options:
  • diff -u file1 file2 - Unified format (easier to read)
  • diff -y file1 file2 - Side-by-side comparison
  • diff -r dir1 dir2 - Recursively compare directories

‚ö° Level 18 ‚Üí 19: Bypassing Auto-Logout (SSH Command Injection)

What is .bashrc?

.bashrc is a script that runs EVERY time you start a bash shell. It sets up your environment (aliases, colors, prompt, etc.)

The problem: Someone modified .bashrc to run exit, so you get logged out instantly!

The trick: You can run commands through SSH WITHOUT starting an interactive shell! Syntax: ssh user@host "command"

‚úÖ Step-by-Step Solution

  1. Try normal login (fails):
    ssh bandit18@bandit.labs.overthewire.org -p 2220
    Immediately disconnected!
  2. Run command directly through SSH:
    ssh bandit18@bandit.labs.overthewire.org -p 2220 "cat readme"
    Breaking this down:
    ‚Ä¢ SSH connects
    ‚Ä¢ Runs the command cat readme
    ‚Ä¢ Shows output
    ‚Ä¢ Disconnects
    .bashrc never gets a chance to run!
  3. The password appears!
‚úÖ What You Learned:
  • .bashrc runs on every bash shell startup
  • ssh user@host "command" runs non-interactive command
  • You can bypass shell initialization this way
  • This technique works for restricted shells too!
üí° Alternative Solutions:
  • ssh ... "bash --norc" - Start bash without .bashrc
  • ssh ... "bash --noprofile" - Skip profile scripts
  • ssh ... "/bin/sh" - Use different shell (sh doesn't use .bashrc)

üëë Level 19 ‚Üí 20: SETUID Privilege Escalation

üéØ GOAL:

Use the setuid binary to access bandit20's password
ssh bandit19@bandit.labs.overthewire.org -p 2220

üìö Concepts You Need to Know

What is SETUID?

SETUID (Set User ID) is a special permission bit. When you run a SETUID program, it executes with the permissions of the FILE OWNER, not YOU!

Real-world example: The passwd command (changes your password) is SETUID root. Why? Because it needs to edit /etc/shadow (a root-only file), but YOU need to be able to change YOUR password!

How to spot SETUID: Run ls -l file. If you see an 's' in the execute position: -rwsr-xr-x (the 's' means SETUID!)

Security risk: SETUID binaries are GOLD for hackers! If you can exploit one, you get higher privileges (privilege escalation!)

‚úÖ Step-by-Step Solution

  1. List files:
    ls -la
    You see: -rwsr-x--- 1 bandit20 bandit19 bandit20-do
    Notice the 's'! This is SETUID and owned by bandit20!
  2. Test the binary:
    ./bandit20-do
    Output: Run a command as another user.
  3. Check who you are normally:
    whoami
    Output: bandit19
  4. Use the SETUID binary to run commands as bandit20:
    ./bandit20-do cat /etc/bandit_pass/bandit20
    What happens:
    ‚Ä¢ The binary runs as bandit20 (owner)
    ‚Ä¢ It executes: cat /etc/bandit_pass/bandit20
    ‚Ä¢ Since it's running AS bandit20, it can read bandit20's password!
  5. Password appears!
‚úÖ What You Learned:
  • SETUID programs run with owner's permissions
  • ls -l shows 's' for SETUID files
  • This is a common privilege escalation vector
  • whoami shows current user
  • SETUID is powerful but dangerous!
üí° Finding SETUID Files (Hacking):
  • find / -perm -4000 2>/dev/null - Find all SETUID files
  • find / -user root -perm -4000 2>/dev/null - SETUID root files (highest value!)
  • Check GTFOBins for exploitable SETUID binaries

üîå Level 20 ‚Üí 21: Network Listener & Background Processes

üéØ GOAL:

SETUID binary suconnect connects to localhost on port you specify
If it receives current password, it transmits next password
ssh bandit20@bandit.labs.overthewire.org -p 2220

üìö Concepts You Need to Know

The Challenge:

You need TWO things running AT THE SAME TIME:

  • Server: A listener that sends the current password when connected to
  • Client: The suconnect binary that connects to your listener

Background processes: Adding & at the end of a command runs it in the background, so you can run more commands!

‚úÖ Step-by-Step Solution

  1. Get current password:
    cat /etc/bandit_pass/bandit20
    Copy it!
  2. Start a listener on any port (e.g., 6000):
    echo "[PASTE BANDIT20 PASSWORD]" | nc -l -p 6000 &
    Breaking this down:
    ‚Ä¢ echo "password" - Output the password
    ‚Ä¢ | - Pipe it to netcat
    ‚Ä¢ nc -l -p 6000 - Listen on port 6000
    ‚Ä¢ & - Run in background!
  3. Verify it's listening:
    jobs
    Or: netstat -ln | grep 6000
  4. Now connect with suconnect:
    ./suconnect 6000
    What happens:
    ‚Ä¢ suconnect connects to port 6000
    ‚Ä¢ Your listener sends the bandit20 password
    ‚Ä¢ suconnect verifies it's correct
    ‚Ä¢ suconnect sends back bandit21 password!
  5. Output:
    Read: [BANDIT20 PASSWORD]
    Password matches, sending next password
    [BANDIT21 PASSWORD]
‚úÖ What You Learned:
  • & runs commands in background
  • nc -l -p PORT creates a listener
  • jobs shows background processes
  • You can pipe data to network connections
  • Client-server communication basics
üí° Process Control:
  • command & - Run in background
  • jobs - List background jobs
  • fg %1 - Bring job 1 to foreground
  • bg %1 - Continue job 1 in background
  • Ctrl+Z - Suspend current process
  • kill %1 - Kill job 1
