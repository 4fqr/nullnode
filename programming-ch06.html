Full-Stack Philosophy

You've learned to program. Now learn to architect. Full-stack development isn't just knowing two technologies‚Äîit's understanding how systems work together, how data flows, how users interact with interfaces, and how servers process their requests. This is where you become an engineer.

The Complete Web Application

When a user types a URL and hits Enter, an intricate dance begins:

  1. Frontend Renders: HTML/CSS/JavaScript creates the visual interface users see and interact with
  2. User Interacts: Click button, submit form, scroll page - JavaScript responds in real-time
  3. API Request: Frontend sends HTTP request to backend server (POST, GET, PUT, DELETE)
  4. Backend Processes: Server validates input, queries database, runs business logic
  5. Database Query: SQL executes, data retrieved/modified, transactions managed
  6. Response Sent: Backend formats data (usually JSON) and sends back to frontend
  7. Frontend Updates: JavaScript receives data, updates DOM, user sees result

Full-stack developers understand every step of this process. When something breaks, they can debug it. When performance lags, they know where to optimize. When requirements change, they architect solutions that span the entire stack.

The Restaurant Metaphor

Frontend is the dining room - beautiful design, friendly interface, where customers interact with your business. A waiter (JavaScript) takes orders and delivers food.

Backend is the kitchen - where real work happens. Chefs (business logic) prepare dishes, ingredients come from storage (database), quality control ensures everything is safe.

API is the window between kitchen and dining room - structured communication protocol where orders go in, food comes out.

A full-stack developer is the restaurant owner who understands both front-of-house and back-of-house. They know why customers complain AND how to fix kitchen inefficiencies.

Why Frontend and Backend Are Different Worlds

These aren't just two sides of the same coin‚Äîthey're fundamentally different programming paradigms:

Aspect Frontend Backend
Environment Runs in browser (user's machine) Runs on server (your infrastructure)
Language JavaScript (no choice) Python, Ruby, Go, Java, etc.
Primary Concern User experience, responsiveness, design Data integrity, security, scalability
State Management Complex (user interactions, async updates) Simpler (request-response cycle)
Performance Focus Minimize bundle size, optimize rendering Database queries, concurrent requests
Security Model Assume everything is visible and manipulable Trust nothing from client, validate everything
Testing Browser compatibility, visual regression Unit tests, integration tests, load tests

The 70/30 Reality

Most full-stack developers lean one direction. Some are frontend developers who can write backend code. Others are backend developers who can style a button.

You'll likely be 70/30 in one direction based on what you enjoy:

  • Frontend-leaning: Love design, animations, user experience, immediate visual feedback
  • Backend-leaning: Love data structures, algorithms, architecture, optimization problems

Both are valuable. The key is being competent enough in both to ship complete features independently.

React Deep Dive

React isn't just a library‚Äîit's a new way of thinking about user interfaces. Forget jQuery and DOM manipulation. React is declarative: you describe what the UI should look like, and React figures out how to make it happen. This is the paradigm shift that changed frontend development forever.

Why React Won

In 2013, React introduced ideas that seemed crazy: JSX (HTML in JavaScript?!), virtual DOM (rebuild the entire UI on every change?!), and components everywhere. Today, these ideas are industry standard. Here's why:

  • Declarative: Describe UI as a function of state, not a series of mutations
  • Component-Based: Build once, reuse everywhere. Encapsulate logic and styling.
  • Learn Once, Write Anywhere: React Native for mobile, React for web, same concepts
  • Massive Ecosystem: Every problem solved, every library integrated, every question answered
  • Industry Adoption: Facebook, Netflix, Airbnb, Instagram - React runs the internet

The Blueprint Metaphor

Imperative (jQuery era): "Go to the second floor, third room on the left, paint the wall blue, then move the chair to the corner."

Declarative (React era): "Here's a blueprint of how the room should look." React figures out what needs to change to match the blueprint.

You describe the destination, not the journey. React handles the journey optimally.

React Fundamentals: Components

Everything in React is a component. A button is a component. A form is a component. An entire page is a component. Components compose‚Äîbig components contain small components.

Function Components (Modern React)

The simplest component is a JavaScript function that returns JSX:

// Simple component function Welcome() { return <h1>Hello, World!</h1>; } // Component with props (data passed from parent) function Greeting({ name, age }) { return ( <div> <h1>Hello, {name}!</h1> <p>You are {age} years old.</p> </div> ); } // Using components function App() { return ( <div> <Welcome /> <Greeting name="Alice" age={25} /> <Greeting name="Bob" age={30} /> </div> ); }

What's happening:

  • JSX: Looks like HTML, but it's JavaScript. Compiles to React.createElement() calls
  • Props: Data passed down from parent components (read-only, immutable)
  • Composition: App component contains Welcome and Greeting components
  • Curly Braces: {name} means "evaluate this JavaScript expression"

JSX Rules You Must Know

  • Return single element: Wrap multiple elements in <div> or <>...</> fragment
  • Close all tags: <img /> not <img>, <br /> not <br>
  • className not class: <div className="card"> (class is JavaScript keyword)
  • camelCase attributes: onClick not onclick, backgroundColor not background-color
  • JavaScript in curly braces: {2 + 2}, {user.name}, {isLoggedIn ? "Logout" : "Login"}

State: Making Components Interactive

Props are data from parent. State is data that changes over time. When state changes, React re-renders the component. This is the magic of React.

The useState Hook

Hooks are functions that let you "hook into" React features. useState is the most important:

import { useState } from 'react'; function Counter() { // Declare state variable const [count, setCount] = useState(0); return ( <div> <p>Count: {count}</p> <button onClick={() => setCount(count + 1)}> Increment </button> <button onClick={() => setCount(count - 1)}> Decrement </button> <button onClick={() => setCount(0)}> Reset </button> </div> ); }

Breakdown:

  • useState(0) - Initialize state to 0
  • count - Current value of state
  • setCount - Function to update state (triggers re-render)
  • onClick - Event handler (React wraps DOM events)
  • Every setCount call causes component to re-render with new value

More Complex State: Objects and Arrays

State can hold any JavaScript value‚Äîobjects, arrays, nested structures:

function TodoList() { const [todos, setTodos] = useState([]); const [inputValue, setInputValue] = useState(''); const addTodo = () => { if (inputValue.trim()) { setTodos([...todos, { id: Date.now(), text: inputValue, completed: false }]); setInputValue(''); } }; const toggleTodo = (id) => { setTodos(todos.map(todo => todo.id === id ? { ...todo, completed: !todo.completed } : todo )); }; const deleteTodo = (id) => { setTodos(todos.filter(todo => todo.id !== id)); }; return ( <div> <h2>Todo List</h2> <input value={inputValue} onChange={(e) => setInputValue(e.target.value)} onKeyPress={(e) => e.key === 'Enter' && addTodo()} placeholder="Add a todo..." /> <button onClick={addTodo}>Add</button> <ul> {todos.map(todo => ( <li key={todo.id}> <input type="checkbox" checked={todo.completed} onChange={() => toggleTodo(todo.id)} /> <span style={{ textDecoration: todo.completed ? 'line-through' : 'none' }}> {todo.text} </span> <button onClick={() => deleteTodo(todo.id)}> Delete </button> </li> ))} </ul> </div> ); }

Critical concepts here:

  • Immutability: Never mutate state directly (todos.push() ‚ùå). Always create new arrays/objects ([...todos, newTodo] ‚úÖ)
  • Array methods: map, filter, find return new arrays (good for React)
  • Keys: key={todo.id} helps React identify which items changed (performance + correctness)
  • Controlled inputs: value and onChange make React control the input (single source of truth)

Why Immutability Matters

React compares old state to new state to decide if re-render is needed. If you mutate the array, the reference is the same, so React thinks nothing changed:

// ‚ùå BAD - mutates existing array todos.push(newTodo); setTodos(todos); // React doesn't detect change! // ‚úÖ GOOD - creates new array setTodos([...todos, newTodo]); // React sees new reference, re-renders

Rule: Treat state as immutable. Create new objects/arrays instead of modifying existing ones.

Side Effects: The useEffect Hook

Components often need to interact with the outside world: fetch data from APIs, update document title, set up subscriptions, manage timers. These are side effects.

useEffect Basics

The second most important hook:

import { useState, useEffect } from 'react'; function UserProfile({ userId }) { const [user, setUser] = useState(null); const [loading, setLoading] = useState(true); const [error, setError] = useState(null); useEffect(() => { // This code runs after component renders setLoading(true); fetch(`https://api.example.com/users/${userId}`) .then(response => response.json()) .then(data => { setUser(data); setLoading(false); }) .catch(err => { setError(err.message); setLoading(false); }); }, [userId]); // Re-run effect when userId changes if (loading) return <p>Loading...</p>; if (error) return <p>Error: {error}</p>; if (!user) return <p>No user found</p>; return ( <div> <h2>{user.name}</h2> <p>Email: {user.email}</p> <p>Bio: {user.bio}</p> </div> ); }

useEffect anatomy:

  • First argument: Function containing side effect code (runs after render)
  • Second argument: Dependency array‚Äîeffect re-runs when these values change
  • Empty array []: Effect runs once on mount (like componentDidMount)
  • No array: Effect runs after every render (usually wrong!)

Cleanup Functions

Some effects need cleanup (remove event listeners, cancel subscriptions, clear timers):

function Timer() { const [seconds, setSeconds] = useState(0); useEffect(() => { // Set up interval const interval = setInterval(() => { setSeconds(s => s + 1); }, 1000); // Cleanup function (runs when component unmounts or before re-running effect) return () => { clearInterval(interval); }; }, []); // Empty array = run once on mount return <p>Seconds elapsed: {seconds}</p>; }

Why cleanup matters: Without clearing the interval, it keeps running even after component is removed from the page. Memory leak! Cleanup prevents this.

Common useEffect Patterns

// 1. Fetch data on mount useEffect(() => { fetch('/api/data').then(/* ... */); }, []); // 2. Sync with external system useEffect(() => { document.title = `Count: ${count}`; }, [count]); // 3. Subscribe to events useEffect(() => { const handleResize = () => setWidth(window.innerWidth); window.addEventListener('resize', handleResize); return () => window.removeEventListener('resize', handleResize); }, []); // 4. Debounce input useEffect(() => { const timer = setTimeout(() => { // API call with searchTerm }, 500); return () => clearTimeout(timer); }, [searchTerm]);

Custom Hooks: Reusable Logic

Hooks are just functions. You can create your own to extract and reuse stateful logic:

useFetch - Reusable Data Fetching

function useFetch(url) { const [data, setData] = useState(null); const [loading, setLoading] = useState(true); const [error, setError] = useState(null); useEffect(() => { setLoading(true); setError(null); fetch(url) .then(res => res.json()) .then(data => { setData(data); setLoading(false); }) .catch(err => { setError(err.message); setLoading(false); }); }, [url]); return { data, loading, error }; } // Usage in components function UserList() { const { data: users, loading, error } = useFetch('/api/users'); if (loading) return <p>Loading...</p>; if (error) return <p>Error: {error}</p>; return ( <ul> {users.map(user => ( <li key={user.id}>{user.name}</li> ))} </ul> ); }

Custom hook benefits:

  • Extract complex logic into reusable functions
  • Share stateful logic between components (without render props or HOCs)
  • Make components cleaner and easier to test
  • Build your own React utilities library

More Useful Custom Hooks

// useLocalStorage - Persist state in browser function useLocalStorage(key, initialValue) { const [value, setValue] = useState(() => { const stored = localStorage.getItem(key); return stored ? JSON.parse(stored) : initialValue; }); useEffect(() => { localStorage.setItem(key, JSON.stringify(value)); }, [key, value]); return [value, setValue]; } // useDebounce - Delay updates until user stops typing function useDebounce(value, delay) { const [debouncedValue, setDebouncedValue] = useState(value); useEffect(() => { const timer = setTimeout(() => setDebouncedValue(value), delay); return () => clearTimeout(timer); }, [value, delay]); return debouncedValue; } // useWindowSize - Track window dimensions function useWindowSize() { const [size, setSize] = useState({ width: window.innerWidth, height: window.innerHeight }); useEffect(() => { const handleResize = () => { setSize({ width: window.innerWidth, height: window.innerHeight }); }; window.addEventListener('resize', handleResize); return () => window.removeEventListener('resize', handleResize); }, []); return size; }

Hooks as Lego Blocks

Built-in hooks (useState, useEffect) are basic Lego blocks. Custom hooks are specialized pieces you build by combining basic blocks.

Just as you can build complex Lego structures by connecting simple blocks, you can build complex React applications by combining simple hooks into more powerful custom hooks.

The best React developers have a library of custom hooks they reuse across projects‚Äîauthentication, forms, data fetching, animations, local storage, and more.

React Best Practices (2026)

  • Keep components small: One responsibility per component (50-100 lines max)
  • Lift state up: State should live in the lowest common ancestor of components that need it
  • Use TypeScript: Catch bugs at compile time, better autocomplete, safer refactoring
  • Avoid premature optimization: React is fast. Profile before optimizing (React DevTools Profiler)
  • Compose, don't couple: Small, reusable components > large, complex ones
  • Test user behavior: Use React Testing Library (test what users see, not implementation)

Vue.js Essentials

Vue is React's friendlier cousin‚Äîeasier to learn, more intuitive, with better documentation. While React says "here's JavaScript, figure it out," Vue says "here's a structure, follow it." Both approaches have merit. Let's learn Vue to understand the alternative paradigm.

React is like a professional chef's kitchen‚Äîpowerful tools everywhere, but you need to know what you're doing. Total freedom to create any dish, but also easy to make a mess.

Vue Basics: Single File Components

Vue's signature feature is the Single File Component‚Äîtemplate, script, and styles in one file:

<template> <div class="counter"> <h2>Count: {{ count }}</h2> <button @click="increment">+1</button> <button @click="decrement">-1</button> <button @click="reset">Reset</button> </div> </template> <script setup> import { ref } from 'vue'; // Reactive state const count = ref(0); // Methods function increment() { count.value++; } function decrement() { count.value--; } function reset() { count.value = 0; } </script> <style scoped> .counter { padding: 2rem; text-align: center; } button { margin: 0 0.5rem; padding: 0.5rem 1rem; font-size: 1rem; } </style>

What's happening:

  • <template>: HTML-like syntax, double curly braces for data binding
  • <script setup>: Composition API (modern Vue), cleaner than Options API
  • ref(): Makes value reactive (Vue tracks changes automatically)
  • @click: Shorthand for v-on:click (event handling)
  • <style scoped>: CSS only applies to this component (no global pollution!)

Vue Reactivity: The Magic Behind the Scenes

Vue's reactivity system is its superpower. When you change data, UI updates automatically‚Äîno manual re-rendering needed.

ref vs reactive

Two ways to create reactive state:

<script setup> import { ref, reactive } from 'vue'; // ==================== ref() - Single values ==================== // Use .value to access/modify const count = ref(0); const name = ref('Alice'); const isActive = ref(true); count.value++; // Modify console.log(count.value); // Access // ==================== reactive() - Objects ==================== // No .value needed, but only works with objects const user = reactive({ name: 'Alice', age: 25, email: 'alice@example.com' }); user.age++; // Modify directly console.log(user.name); // Access directly // ==================== When to use which? ==================== // ref() - Primitive values (numbers, strings, booleans) const score = ref(0); const message = ref('Hello'); // reactive() - Complex objects with multiple properties const form = reactive({ username: '', password: '', rememberMe: false }); // ref() can also hold objects, but you need .value const userRef = ref({ name: 'Bob', age: 30 }); userRef.value.age++; // Need .value to access object </script>

Pro Tip: ref() for Everything

Many Vue developers use ref() for everything (even objects) because it's consistent. Yes, you need .value, but you always know when you're dealing with reactive data.

reactive() is great for objects, but has limitations: can't be reassigned, loses reactivity when destructured, doesn't work with primitives.

Modern best practice: Use ref() everywhere unless you have a specific reason to use reactive().

Vue Directives: Template Superpowers

Directives are special HTML attributes that give your templates dynamic behavior:

<template> <div> <!-- v-if - Conditional rendering (removes from DOM) --> <p v-if="score >= 90">Excellent!</p> <p v-else-if="score >= 70">Good job!</p> <p v-else>Keep practicing!</p> <!-- v-show - Toggle visibility (display: none) --> <div v-show="isVisible">I can be toggled!</div> <!-- v-for - Loop over arrays --> <ul> <li v-for="(item, index) in items" :key="item.id"> {{ index }}: {{ item.name }} </li> </ul> <!-- v-bind (shorthand :) - Bind attributes --> <img :src="imageUrl" :alt="imageAlt" /> <a :href="linkUrl">{{ linkText }}</a> <div :class="{ active: isActive, disabled: !isEnabled }"></div> <!-- v-on (shorthand @) - Event handlers --> <button @click="handleClick">Click Me</button> <input @keyup.enter="submitForm" /> <form @submit.prevent="onSubmit">...</form> <!-- v-model - Two-way binding --> <input v-model="username" placeholder="Enter username" /> <textarea v-model="bio"></textarea> <input type="checkbox" v-model="agreeToTerms" /> <select v-model="selectedCountry"> <option value="us">United States</option> <option value="uk">United Kingdom</option> </select> </div> </template> <script setup> import { ref } from 'vue'; const score = ref(85); const isVisible = ref(true); const items = ref([ { id: 1, name: 'Apple' }, { id: 2, name: 'Banana' }, { id: 3, name: 'Cherry' } ]); const imageUrl = ref('/logo.png'); const imageAlt = ref('Company Logo'); const linkUrl = ref('https://example.com'); const linkText = ref('Visit Example'); const isActive = ref(true); const isEnabled = ref(false); const username = ref(''); const bio = ref(''); const agreeToTerms = ref(false); const selectedCountry = ref('us'); function handleClick() { console.log('Button clicked!'); } function submitForm() { console.log('Form submitted!'); } function onSubmit() { console.log('Submit prevented, custom logic runs'); } </script>

Computed Properties and Watchers

Sometimes you need derived state‚Äîvalues calculated from other state:

<template> <div> <input v-model="firstName" placeholder="First name" /> <input v-model="lastName" placeholder="Last name" /> <p>Full name: {{ fullName }}</p> <p>Initials: {{ initials }}</p> <hr> <input v-model.number="price" type="number" /> <input v-model.number="quantity" type="number" /> <p>Subtotal: ${{ subtotal }}</p> <p>Tax (10%): ${{ tax }}</p> <p>Total: ${{ total }}</p> </div> </template> <script setup> import { ref, computed, watch } from 'vue'; const firstName = ref(''); const lastName = ref(''); const price = ref(0); const quantity = ref(1); // Computed properties - cached, only recalculate when dependencies change const fullName = computed(() => { return `${firstName.value} ${lastName.value}`.trim(); }); const initials = computed(() => { const first = firstName.value[0] || ''; const last = lastName.value[0] || ''; return `${first}${last}`.toUpperCase(); }); const subtotal = computed(() => price.value * quantity.value); const tax = computed(() => subtotal.value * 0.1); const total = computed(() => subtotal.value + tax.value); // Watchers - run side effects when values change watch(fullName, (newName, oldName) => { console.log(`Name changed from "${oldName}" to "${newName}"`); }); watch(total, (newTotal) => { if (newTotal > 100) { console.log('High value order! Consider offering free shipping.'); } }); </script>

computed vs methods:

  • Computed: Cached based on dependencies. Only recalculates when dependencies change. Use for expensive calculations.
  • Methods: Always execute when called. Use for actions/events that should run every time.

Vue's Killer Features

  • Scoped styles: CSS only affects current component (no naming conflicts!)
  • v-model: Two-way binding is dead simple (React requires onChange + value)
  • Transitions: Built-in <Transition> component for animations
  • Better TypeScript: Vue 3 written in TypeScript, excellent type inference
  • Documentation: Objectively the best docs in frontend (React's are... improving)
  • Learning curve: HTML/CSS developers can be productive in days, not weeks

When to Choose Vue Over React

‚úÖ Choose Vue When:

  • You're new to frontend frameworks
  • You prefer HTML templates over JSX
  • You want official state management (Vuex/Pinia)
  • You're a solo developer or small team
  • You want excellent documentation

‚úÖ Choose React When:

  • You're comfortable with JavaScript
  • You need the largest ecosystem
  • You want maximum flexibility
  • You're joining a large team/company
  • You want React Native for mobile

The truth: Both are excellent. Learn React first (more jobs), then pick up Vue in a weekend (concepts transfer instantly). Knowing both makes you more employable and helps you choose the right tool for each project.

Component Architecture

Components are the building blocks of modern web apps. But there's an art to designing them‚Äîtoo big and they're unmaintainable, too small and you drown in abstractions. Let's learn to architect component hierarchies like professionals.

The Component Mindset

Look at any website. Now mentally break it down:

  • Page - The top-level component
  • Layout - Header, sidebar, main content, footer
  • Sections - Hero, features, pricing, testimonials
  • Cards - Individual feature cards, user profiles
  • Elements - Buttons, inputs, icons, labels

Everything is a component. The question is: where do you draw the boundaries?

The Lego Analogy (Again)

Good components are like Lego blocks:

  • Single purpose: A wheel is a wheel, not a wheel-plus-engine
  • Reusable: Same wheel fits many vehicles
  • Composable: Blocks connect to form larger structures
  • Self-contained: Block works independently, doesn't require specific neighbors

Component Design Principles

1. Single Responsibility Principle

Each component should do ONE thing well:

// ‚ùå BAD - Too many responsibilities function UserDashboard() { const [user, setUser] = useState(null); const [posts, setPosts] = useState([]); const [comments, setComments] = useState([]); const [followers, setFollowers] = useState([]); const [notifications, setNotifications] = useState([]); // 200 lines of mixed logic... return ( <div> {/* Complex UI mixing user profile, posts, comments, etc. */} </div> ); } // ‚úÖ GOOD - Split into focused components function UserDashboard() { return ( <div> <UserProfile /> <PostList /> <CommentSection /> <FollowersList /> <NotificationBell /> </div> ); } function UserProfile() { const [user, setUser] = useState(null); // Logic only for user profile } function PostList() { const [posts, setPosts] = useState([]); // Logic only for posts }

2. Props Down, Events Up

Data flows down through props. Events bubble up through callbacks:

// Parent component function TodoApp() { const [todos, setTodos] = useState([]); const addTodo = (text) => { setTodos([...todos, { id: Date.now(), text, done: false }]); }; const toggleTodo = (id) => { setTodos(todos.map(t => t.id === id ? { ...t, done: !t.done } : t )); }; return ( <div> <TodoInput onAdd={addTodo} /> {/* Event up */} <TodoList todos={todos} {/* Data down */} onToggle={toggleTodo} {/* Event up */} /> </div> ); } // Child components function TodoInput({ onAdd }) { const [text, setText] = useState(''); const handleSubmit = () => { if (text.trim()) { onAdd(text); // Event up to parent setText(''); } }; return ( <input value={text} onChange={(e) => setText(e.target.value)} onKeyPress={(e) => e.key === 'Enter' && handleSubmit()} /> ); } function TodoList({ todos, onToggle }) { return ( <ul> {todos.map(todo => ( <TodoItem key={todo.id} todo={todo} {/* Data down */} onToggle={onToggle} {/* Event up */} /> ))} </ul> ); } function TodoItem({ todo, onToggle }) { return ( <li onClick={() => onToggle(todo.id)}> {/* Event up */} {todo.done ? '‚úì' : '‚óã'} {todo.text} </li> ); }

Why this pattern? State lives in one place (parent), components are pure functions of props, data flow is predictable.

3. Composition Over Configuration

Instead of complex props, compose components:

// ‚ùå BAD - Props explosion function Card({ title, subtitle, body, footer, hasImage, imageUrl, imageAlt, onClick, className, // ...20 more props }) { // Complex conditional logic based on props } // ‚úÖ GOOD - Composition function Card({ children, onClick, className }) { return ( <div className={`card ${className}`} onClick={onClick}> {children} </div> ); } // Usage - total flexibility! <Card> <CardImage src="/photo.jpg" alt="Photo" /> <CardHeader> <h2>Title</h2> <p>Subtitle</p> </CardHeader> <CardBody> <p>Any content here!</p> </CardBody> <CardFooter> <Button>Action</Button> </CardFooter> </Card>

4. Presentational vs Container Components

Separate how things look from how things work:

Component Patterns

Pattern 1: Compound Components

Components that work together, sharing implicit state:

// Design goal: intuitive API <Tabs defaultTab="profile"> <TabList> <Tab id="profile">Profile</Tab> <Tab id="settings">Settings</Tab> <Tab id="notifications">Notifications</Tab> </TabList> <TabPanels> <TabPanel id="profile">Profile content...</TabPanel> <TabPanel id="settings">Settings content...</TabPanel> <TabPanel id="notifications">Notifications content...</TabPanel> </TabPanels> </Tabs> // Implementation using Context const TabsContext = React.createContext(); function Tabs({ defaultTab, children }) { const [activeTab, setActiveTab] = useState(defaultTab); return ( <TabsContext.Provider value={{ activeTab, setActiveTab }}> <div className="tabs">{children}</div> </TabsContext.Provider> ); } function Tab({ id, children }) { const { activeTab, setActiveTab } = useContext(TabsContext); return ( <button className={activeTab === id ? 'active' : ''} onClick={() => setActiveTab(id)} > {children} </button> ); } function TabPanel({ id, children }) { const { activeTab } = useContext(TabsContext); return activeTab === id ? <div>{children}</div> : null; }

Pattern 2: Render Props

Pass rendering logic as a prop:

// Reusable data fetching component function DataFetcher({ url, render }) { const [data, setData] = useState(null); const [loading, setLoading] = useState(true); const [error, setError] = useState(null); useEffect(() => { fetch(url) .then(res => res.json()) .then(data => { setData(data); setLoading(false); }) .catch(err => { setError(err.message); setLoading(false); }); }, [url]); return render({ data, loading, error }); } // Usage <DataFetcher url="/api/users" render={({ data, loading, error }) => { if (loading) return <Spinner />; if (error) return <Error message={error} />; return ( <ul> {data.map(user => <li key={user.id}>{user.name}</li>)} </ul> ); }} />

Pattern 3: Higher-Order Components (HOCs)

Functions that take a component and return an enhanced component:

// HOC that adds authentication check function withAuth(Component) { return function AuthenticatedComponent(props) { const { user, loading } = useAuth(); if (loading) return <Spinner />; if (!user) return <Redirect to="/login" />; return <Component {...props} user={user} />; }; } // Usage const Dashboard = ({ user }) => { return <div>Welcome, {user.name}!</div>; }; export default withAuth(Dashboard); // Enhanced with auth logic

Modern Pattern Preference

With hooks, many patterns are less necessary:

  • Custom hooks > HOCs: Hooks are simpler, more flexible, easier to understand
  • Custom hooks > Render props: Cleaner component code, better TypeScript support
  • Compound components: Still great for complex UI components (Tabs, Accordions, Modals)

But know all patterns‚Äîyou'll see them in existing codebases!

Flask Backend

Flask is Python's micro-framework‚Äîminimalist, flexible, perfect for learning. It gives you the essentials (routing, request handling, templates) and gets out of your way. If React is "just JavaScript," Flask is "just Python." Let's build APIs.

Why Flask for Backend Development

Django is the full-featured framework. Flask is the lightweight one. Here's why Flask is excellent for learning:

  • Minimal boilerplate: 5 lines for "Hello World" server
  • Explicit: You see exactly what's happening (no magic)
  • Flexible: Pick your database, templating, auth system
  • Perfect for APIs: RESTful API in minutes
  • Great documentation: Clear, concise, well-written

The Toolkit Metaphor

Flask is like a basic toolkit‚Äîhammer, screwdriver, wrench. You choose what to build and how. Flexible, but you make all decisions.

Django is like a pre-assembled workstation‚Äîeverything included, organized, optimized. Fast to use if you work their way, constraining if you don't.

Learn Flask first to understand how frameworks work. Graduate to Django when you need batteries included.

Building a REST API

Let's build a complete CRUD API for managing tasks:

from flask import Flask, request, jsonify from datetime import datetime app = Flask(__name__) # In-memory storage (in real app, use database) tasks = [] next_id = 1 # ==================== CREATE ==================== @app.route('/api/tasks', methods=['POST']) def create_task(): global next_id data = request.json # Validation if not data or 'title' not in data: return jsonify({'error': 'Title is required'}), 400 task = { 'id': next_id, 'title': data['title'], 'description': data.get('description', ''), 'completed': False, 'created_at': datetime.now().isoformat() } tasks.append(task) next_id += 1 return jsonify(task), 201 # ==================== READ (LIST) ==================== @app.route('/api/tasks', methods=['GET']) def get_tasks(): # Optional filtering completed = request.args.get('completed', type=bool) if completed is not None: filtered = [t for t in tasks if t['completed'] == completed] return jsonify(filtered) return jsonify(tasks) # ==================== READ (SINGLE) ==================== @app.route('/api/tasks/<int:task_id>', methods=['GET']) def get_task(task_id): task = next((t for t in tasks if t['id'] == task_id), None) if not task: return jsonify({'error': 'Task not found'}), 404 return jsonify(task) # ==================== UPDATE ==================== @app.route('/api/tasks/<int:task_id>', methods=['PUT']) def update_task(task_id): task = next((t for t in tasks if t['id'] == task_id), None) if not task: return jsonify({'error': 'Task not found'}), 404 data = request.json # Update fields if 'title' in data: task['title'] = data['title'] if 'description' in data: task['description'] = data['description'] if 'completed' in data: task['completed'] = data['completed'] task['updated_at'] = datetime.now().isoformat() return jsonify(task) # ==================== DELETE ==================== @app.route('/api/tasks/<int:task_id>', methods=['DELETE']) def delete_task(task_id): global tasks task = next((t for t in tasks if t['id'] == task_id), None) if not task: return jsonify({'error': 'Task not found'}), 404 tasks = [t for t in tasks if t['id'] != task_id] return jsonify({'message': 'Task deleted'}), 200 # ==================== ERROR HANDLERS ==================== @app.errorhandler(404) def not_found(error): return jsonify({'error': 'Not found'}), 404 @app.errorhandler(500) def internal_error(error): return jsonify({'error': 'Internal server error'}), 500 if __name__ == '__main__': app.run(debug=True)

Flask Best Practices

  • Use Blueprints: Organize routes into modules (auth routes, api routes, etc.)
  • Environment variables: Config settings (database URLs, secret keys) in .env files
  • Application factory: Create app in function (better for testing, multiple configs)
  • Error handling: Use @app.errorhandler for consistent error responses
  • Input validation: Validate ALL user input (never trust client)
  • Use extensions: Don't reinvent wheels (auth, database, migrations)

Django Framework - The Batteries-Included Backend

Django is Python's most comprehensive web framework. Where Flask gives you flexibility, Django gives you everything out of the box - ORM, admin panel, authentication, forms, security. It's opinionated, structured, and production-ready from day one.

Django follows the "batteries included" philosophy. Need user authentication? Built-in. Need an admin interface? Auto-generated. Need database migrations? Handled. Django makes decisions for you so you can focus on building features, not infrastructure.

Key concepts:

  • Primary Key: Unique identifier for each row (usually id)
  • Foreign Key: Reference to another table (author_id points to Users.id)
  • Relationships: One-to-Many (user has many posts), Many-to-Many (posts have many tags)

SQL Basics - Talking to Databases

SELECT - Reading Data

-- Get all users SELECT * FROM users; -- Get specific columns SELECT username, email FROM users; -- Filter with WHERE SELECT * FROM users WHERE id = 1; SELECT * FROM posts WHERE views > 100; -- Multiple conditions SELECT * FROM posts WHERE published = true AND views > 50; -- Pattern matching SELECT * FROM users WHERE email LIKE '%@gmail.com'; -- Sorting SELECT * FROM posts ORDER BY created DESC; SELECT * FROM users ORDER BY username ASC; -- Limit results SELECT * FROM posts ORDER BY views DESC LIMIT 10;

INSERT - Creating Data

-- Insert single row INSERT INTO users (username, email, created) VALUES ('alice', 'alice@example.com', '2024-01-15'); -- Insert multiple rows INSERT INTO posts (title, author_id, views, created) VALUES ('First Post', 1, 0, '2024-01-16'), ('Second Post', 1, 0, '2024-01-18'), ('Third Post', 2, 0, '2024-01-20');

UPDATE - Modifying Data

-- Update single field UPDATE posts SET views = views + 1 WHERE id = 1; -- Update multiple fields UPDATE users SET email = 'newemail@example.com', updated = NOW() WHERE id = 1; -- Update based on condition UPDATE posts SET published = true WHERE created < '2024-02-01';

DELETE - Removing Data

-- Delete specific row DELETE FROM posts WHERE id = 1; -- Delete based on condition DELETE FROM comments WHERE created < '2023-01-01'; -- Delete all rows (dangerous!) DELETE FROM temp_data;

JOINs - Combining Tables

The power of relational databases - connecting related data:

-- INNER JOIN - Get posts with author info SELECT posts.title, users.username, posts.views FROM posts INNER JOIN users ON posts.author_id = users.id; -- LEFT JOIN - Get all posts, even without author SELECT posts.title, users.username FROM posts LEFT JOIN users ON posts.author_id = users.id; -- Multiple JOINs SELECT posts.title, users.username AS author, COUNT(comments.id) AS comment_count FROM posts INNER JOIN users ON posts.author_id = users.id LEFT JOIN comments ON comments.post_id = posts.id GROUP BY posts.id, users.username;

Database Design Principles

Principle Description Example
Normalization Eliminate data duplication Store author once in users table, reference by ID
Atomic Values Each cell should contain one value Don't store "tag1,tag2,tag3" - use separate tags table
Indexing Speed up queries on frequently searched columns Index email for login queries, created for sorting
Constraints Enforce data integrity UNIQUE on email, NOT NULL on username

Popular Database Choices

SQLite: File-based, perfect for development and small apps. Comes with Python.

PostgreSQL: Most powerful open-source DB. Advanced features, excellent performance. Industry standard.

MySQL/MariaDB: Very popular, fast for reads. Used by WordPress, Facebook.

MongoDB: NoSQL document database. Good for flexible schemas, but loses relational power.

For learning: Start with SQLite. For production: PostgreSQL is the safe choice.

REST APIs - The Language of the Web

APIs are how applications talk to each other. Your React frontend talks to your Django backend through an API. Third-party services integrate with your app through APIs. Understanding REST principles makes you a complete full-stack developer.

What is REST?

REST (Representational State Transfer) is an architectural style for designing networked applications. Key principles:

  • Stateless: Each request contains all information needed - no session state on server
  • Client-Server: Frontend and backend are separate, can evolve independently
  • Resource-Based: Everything is a resource (users, posts, comments) with unique URLs
  • HTTP Methods: Standard methods for CRUD operations

HTTP Methods and Their Meaning

Method Purpose Example Idempotent?
GET Retrieve resource GET /api/posts/1 Yes
POST Create new resource POST /api/posts No
PUT Update/replace resource PUT /api/posts/1 Yes
PATCH Partial update PATCH /api/posts/1 No
DELETE Remove resource DELETE /api/posts/1 Yes

Idempotent means making the same request multiple times has the same effect as making it once. GET always returns same data. DELETE deletes once, subsequent calls do nothing. POST creates new resource each time (not idempotent).

API Design Best Practices

# Good RESTful API design: # Resource collections (plural nouns) GET /api/posts # List all posts POST /api/posts # Create new post # Specific resources (with ID) GET /api/posts/123 # Get post 123 PUT /api/posts/123 # Update post 123 DELETE /api/posts/123 # Delete post 123 # Nested resources GET /api/posts/123/comments # Comments for post 123 POST /api/posts/123/comments # Add comment to post 123 # Filtering and pagination GET /api/posts?published=true&limit=10&offset=20 GET /api/posts?author=alice&sort=-created # Search GET /api/posts/search?q=django # Versioning GET /api/v1/posts GET /api/v2/posts

‚úÖ Good API Design

  • Consistent naming (plural nouns)
  • Proper HTTP status codes
  • Comprehensive documentation
  • Versioning for breaking changes

  • Verbs in URLs (/getPost, /createUser)
  • Always returning 200 OK

Authentication & Security - Protecting Your App

Security isn't optional. Every web application needs to authenticate users, protect passwords, and prevent attacks. This section covers the essentials of web security from a full-stack perspective.

Authentication Strategies

üç™ Session-Based Auth

How it works: Server stores session data, sends cookie to client

Pros: Simple, server controls sessions, easy to revoke

Cons: Not scalable, doesn't work for mobile apps

Best for: Traditional web apps with server-side rendering

üé´ Token-Based (JWT)

How it works: Server signs a token, client stores it, includes in requests

Pros: Stateless, scalable, works with mobile/SPA

Cons: Can't revoke without extra infrastructure

Best for: APIs, SPAs, mobile apps

üîë OAuth 2.0

How it works: Third-party handles authentication (Google, GitHub)

Pros: No password storage, user convenience, established trust

Cons: Complex flow, dependency on third party

Best for: Consumer apps, social features

Password Security

Never store passwords in plain text! Always hash them:

# BAD - Never do this! user = {"username": "alice", "password": "secret123"} # GOOD - Use bcrypt from flask_bcrypt import Bcrypt bcrypt = Bcrypt(app) # Hash password on registration hashed = bcrypt.generate_password_hash("secret123").decode('utf-8') user = {"username": "alice", "password_hash": hashed} # Verify password on login if bcrypt.check_password_hash(user['password_hash'], "secret123"): print("Login successful!")

Why bcrypt?

  • Automatically salts passwords (prevents rainbow table attacks)
  • Slow by design (makes brute force impractical)
  • Configurable work factor (increase difficulty over time)

JWT (JSON Web Tokens)

# Install: pip install pyjwt import jwt import datetime SECRET_KEY = 'your-secret-key-here' # Create token on login def create_token(user_id): payload = { 'user_id': user_id, 'exp': datetime.datetime.utcnow() + datetime.timedelta(hours=24), 'iat': datetime.datetime.utcnow() } token = jwt.encode(payload, SECRET_KEY, algorithm='HS256') return token # Verify token on protected routes def verify_token(token): try: payload = jwt.decode(token, SECRET_KEY, algorithms=['HS256']) return payload['user_id'] except jwt.ExpiredSignatureError: return None # Token expired except jwt.InvalidTokenError: return None # Invalid token # Usage in Flask @app.route('/api/protected') def protected(): token = request.headers.get('Authorization') if not token: return jsonify({"error": "No token"}), 401 user_id = verify_token(token) if not user_id: return jsonify({"error": "Invalid token"}), 401 return jsonify({"message": "Access granted", "user": user_id})

Common Security Vulnerabilities

Attack Description Prevention
SQL Injection Malicious SQL in user input Use parameterized queries or ORM
XSS Inject JavaScript in pages Escape user input, use CSP headers
CSRF Forged requests from other sites CSRF tokens, SameSite cookies
Brute Force Guess passwords repeatedly Rate limiting, account lockout, 2FA

Security Best Practices

  • HTTPS everywhere: Never send credentials over HTTP
  • Validate input: Never trust user data
  • Least privilege: Users only get permissions they need
  • Regular updates: Keep dependencies patched
  • Security headers: X-Frame-Options, X-XSS-Protection, CSP
  • Logging: Track failed logins, suspicious activity

State Management in Full-Stack Apps

For complex apps, consider:

  • Context API: Built into React, good for simple state
  • Redux: Centralized store, time-travel debugging, lots of boilerplate
  • Zustand: Simpler than Redux, less boilerplate
  • TanStack Query: Specialized for server state (API data)

Don't over-engineer. Start with useState/useEffect, add complexity only when needed.
